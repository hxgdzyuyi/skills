# Refactor Hotspots

统计 git 仓库中最近一段时间被频繁修改的文件，识别"热点"，然后结合文件内容、修改历史和相关文件给出重构建议。

频繁修改往往说明该处承担了过多责任、抽象层次不当，或存在可提炼的共性——这正是重构最有价值的地方。

## 工作流程

### 1. 确认参数

向用户确认以下两个参数（允许直接回车使用默认值）：

1. **时间范围** — 默认 `1 week ago`。支持 git 识别的任意时间表达（如 `2 weeks ago`、`1 month ago`、`2026-03-01`）
2. **分析的文件数量 N** — 默认 `20`

### 2. 统计热点文件

用户确认后运行（将 `{SINCE}`、`{N}` 替换为实际值）：

```bash
git log --since='{SINCE}' --no-merges --name-only --pretty=format: \
  | awk 'NF' \
  | sort \
  | uniq -c \
  | sort -rn \
  | head -{N}
```

输出每行形如 `  <次数> <文件路径>`。把完整列表展示给用户，便于其对照。

### 3. 排除"代码自动生成"的文件

> 不使用固定的 pattern 黑名单，而是由模型结合文件名、路径、扩展名、必要时查看文件前几十行做判断。原因：不同项目的生成文件惯例差异很大，pattern 容易误伤或漏判。

判断思路：

- **常见强信号**：lock 文件、`dist/`/`build/` 等产物目录、`*.pb.*`/`*_pb2.py`/`*.generated.*` 等命名；文件头部含 `Code generated`、`DO NOT EDIT`、`@generated`、`autogenerated` 等注释
- **弱信号**：目录或文件名看起来像生成物，但不确定。此时读文件头（`head -50`）确认
- **不确定时向用户确认**，而不是擅自排除

列出被排除的文件及各自的排除理由，让用户确认最终要分析的列表（用户可追加或剔除）。

### 4. 逐个文件做重构分析

**对每个待分析的文件**，按下列顺序收集上下文，再输出建议：

1. **读取文件当前内容**

2. **查看在时间窗口内的修改主题**：
   ```bash
   git log --since='{SINCE}' --oneline -- {FILE}
   ```
   必要时再看一段 diff：
   ```bash
   git log --since='{SINCE}' -p -- {FILE} | head -300
   ```
   目标：识别"这段时间为什么在反复改这个文件"——是修 bug、加小 feature，还是抽象边界不对总被迫改？

3. **查看相关文件**（重要 — 单看一个文件容易漏掉跨文件的重构机会）：
   - 该文件 import / require / 引用的模块
   - 与该文件**经常在同一 commit 中被修改**的其它文件（高共变化意味着抽象边界可能错位）：
     ```bash
     # 粗略查看共变关系
     git log --since='{SINCE}' --name-only --pretty=format:'---%n%H' -- {FILE}
     ```
     从结果中观察哪些文件频繁与 `{FILE}` 同现，再对其中最相关的 1-2 个做快读。

4. **产出建议**，至少覆盖这几个视角（不相关的可略过）：
   - **DRY**：是否有重复片段可以抽函数/抽模块？
   - **SRP**：文件/类/函数是否承担了过多责任？是否应当拆分？
   - **频繁修改的根因**：如果同一段代码被反复改，常常意味着抽象不对位——这往往是最值得重构的信号
   - **耦合与边界**：与哪些文件共变？是否应该把共变部分提到公共层？
   - **圈复杂度 / 可读性**：过深嵌套、过长方法、命名晦涩等

### 5. 对话中输出结果

按文件逐个输出，每块用如下结构（Markdown）：

```markdown
### `{文件路径}`（修改 {N} 次）

**近期修改主题**：{一两句话概括}

**重构建议**：
1. {指向具体代码或 commit 的建议}
2. ...

**相关文件**：{分析中用到的相关文件，便于用户顺着看}
```

全部文件分析完成后，追加一段**整体观察**：跨文件反复出现的共性问题、潜在的模块边界调整建议等。

## 原则

- **只分析、不改动**：本 skill 不直接编辑代码，只产出建议
- **每条建议都要有证据**：指向具体代码位置或 commit，不写"可读性不好"这种空话
- **按修改次数排优先级**：若文件较多、时间有限，先把次数最高的几个写透，而不是每个都浅尝
- **如果某个热点文件本质上是配置/文档**（如 CHANGELOG、版本号文件），可以只做一句话说明后跳过
