# 文章轮播卡片

使用这个 Skill 时，让 LLM 直接完成内容整理、版式判断、SVG 编写、预览检查和修正。不要依赖固定生成脚本来替代模型判断。脚本或命令只用于通用检查、文件操作和 `rsvg-convert` 渲染。

## 内容模型

先把任务拆成四类项目：

1. **项目 1：标题或结尾卡片**
   - 通常是预先生成好的 `.png`，例如封面、主标题图、推荐页、关注/订阅引导页。
   - 如果用户说明已经生成，直接复用，并用 `view_image` 查看画面，理解其风格后再生成后续卡片。
   - 如果没有现成图片，且用户需要成品级视觉效果，优先使用图像生成做位图封面/结尾页；如果用户希望全 SVG，则手写 SVG。

2. **项目 2：二级标题卡片**
   - 以标题文字为主导的 SVG 卡片。
   - 语义上相当于文章结构里的二级标题。
   - 可包含来源 URL、日期、作者、分类等短元信息。

3. **项目 3：三级内容卡片**
   - 项目 2 下面的正文解释卡片，仍然使用 SVG。
   - 语义上相当于三级段落。
   - 三级标题可有可无；只有正文的卡片也是合法的。
   - 根据内容需要，可加入嵌入图、示意图、引用块、对比表、强调框等。

4. **项目 4：模板**
   - 项目 2 和项目 3 的 SVG 模板。
   - 文件名固定为 `tmpl_title.svg` 和 `tmpl_content.svg`。
   - 模板应体现当前选定的风格系统，并保持可编辑；模板不计入正式页码。

## 工作流

### 步骤 0：检查

检查环境中是否有 `rsvg-convert`：

```bash
command -v rsvg-convert
```

如果缺失，可以继续创建或编辑 SVG 源文件，但必须明确告诉用户：在安装 librsvg / `rsvg-convert` 前，无法完成 PNG 预渲染和最终渲染验证。

### 步骤 1：整理文本数据源

在生成卡片前，先把用户提供的原始材料整理成下面的概念结构：

- 封面/标题卡片。
- 项目 2：二级标题。
- 项目 3：每个二级标题下的三级内容块。
- 可选的结尾、关注、订阅或行动引导卡片。

当输入较长时，优先在本地临时 Markdown 或 JSON 规划文件中整理成这种结构：

```text
封面:
  标题:
  已有图片:
章节:
  - 二级标题:
    来源链接:
    三级内容卡片:
      - 可选三级标题:
        正文:
        可选视觉元素:
结尾:
  已有图片:
  引导文案:
```

保持结构弹性。有些文章适合一个二级标题配一张内容卡；有些需要多张内容卡；也可能存在只有标题卡、没有后续正文卡的章节。

### 步骤 2：处理项目 1

如果项目 1 图片已经存在：

- 保持图片不变，直接作为有序卡片源文件。
- 使用 `view_image` 查看图片。
- 从图片中提取风格信号：画幅比例、背景气质、字体氛围、主色/强调色、信息密度、品牌元素。

如果项目 1 图片不存在：

- 只有在封面/结尾方向确实不明确时才询问用户。
- 否则根据文章主题和目标平台推断合适方向，并生成位图或手写 SVG。
- 项目 1 必须和其他 SVG 卡片一起纳入最终有序文件序列。

### 步骤 3：生成项目 4 模板

在批量生成卡片前，先创建：

- `tmpl_title.svg`：项目 2 的结构模板。
- `tmpl_content.svg`：项目 3 的结构模板。

模板应沉淀可复用的版式决策：

- 画布尺寸和 `viewBox`。
- 背景色、主色和强调色。
- 字体族、字重、字号、行高。
- 文本安全区域。
- 页眉、页码、页脚规则，如果该风格需要。
- 元信息和来源链接的位置。
- 内容卡片里的图片、引用、强调框等占位规则。

不要写死某一个品牌。只有用户明确要求时才固定品牌；否则根据参考图、文章主题和平台场景自适应。

### 步骤 4：生成项目 2 和项目 3

在输出目录中按顺序创建文件：

```text
001.<short-title>.<svg|png>
002.<short-title>.svg
003.<short-title>.svg
```

规则：

- 使用三位数字编号。
- 文件名保持小写，并避免空格和特殊符号。
- 只统计正式有序卡片；不要把 `tmpl_title.svg`、`tmpl_content.svg`、预览目录、QA 记录计入页数。
- 项目 1 可以是 `.png`；项目 2 和项目 3 通常应为 `.svg`。
- SVG 中的正文尽量使用真实 `<text>` 元素，不要把文字压成不可编辑的位图。
- 只有在可读、可维护时才用 `<tspan>` 做行内强调。
- 长文本需要手动设计换行，不要依赖浏览器自动排版。

### 步骤 5：预渲染

使用 `rsvg-convert` 把 SVG 预渲染到 `_preview/`：

```bash
mkdir -p _preview
find . -maxdepth 1 -type f -name '[0-9][0-9][0-9].*.svg' -print0 |
  while IFS= read -r -d '' f; do
    name="${f#./}"
    rsvg-convert "$f" -o "_preview/${name%.svg}.png"
  done
```

如果项目 1 是现成 PNG，必要时复制到 `_preview/` 方便并排检查。

### 步骤 6：检查

必须检查预览 PNG，而不是只看 SVG 源码。重点检查：

- 禁则：行首不要出现闭合标点，例如 `, . ; : ! ? ) ] } ， 。 、 ； ： ！ ？ ） 】`。
- 行尾不要出现开启标点，例如 `( [ { （ 【`。
- 重要文字不能被裁切，也不能超出安全区域。
- 文字不能和图标、图片、页码、页脚、相邻文字互相遮盖。
- 文本密度要有意识地控制，避免为了塞内容把段落压到难以阅读。
- 嵌入图片要足够清晰，并且不能挤压页眉或页脚。
- 项目 2 的二级标题层级要明显区别于项目 3 的正文卡片。
- 整套卡片从前到后要能连贯阅读。

如果某一页不合格，编辑对应 SVG 后重新预渲染，直到检查通过。

### 步骤 7：正式渲染

预览检查通过后，正式渲染到 `png/`：

```bash
mkdir -p png
find . -maxdepth 1 -type f -name '[0-9][0-9][0-9].*.svg' -print0 |
  while IFS= read -r -d '' f; do
    name="${f#./}"
    rsvg-convert "$f" -o "png/${name%.svg}.png"
  done
```

把有序项目 1 PNG 也复制到 `png/`，并保持相同编号命名。

## 通用适配规则

- 把用户提供的样例当作风格参考，不要当成不可变模板。
- 如果用户提供品牌规范，优先遵守；否则根据主题和平台选择克制、清晰的视觉系统。
- 除非用户明确要求，否则所有有序卡片保持同一画幅比例。
- 项目 2 和项目 3 优先使用可读、可编辑的 SVG，不要为了装饰写过度复杂的图形。
- 只有在项目 1 需要更强视觉表现，或用户明确要求时，才使用位图生成。
- 最终交付物应便于检查：有序源文件、`tmpl_title.svg`、`tmpl_content.svg`、`_preview/`、`png/`，以及必要的 QA 说明。
